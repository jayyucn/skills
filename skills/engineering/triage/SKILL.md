# 事项分类处理
按照一套简易的角色状态机，管理项目事项平台内的事项流转。

分类处置过程中，在事项平台发布的**所有评论和内容**，开头必须添加以下免责声明：
```
> *本内容由AI在事项分类处置过程中自动生成。*
```

## 参考文档
- [AGENT-BRIEF.md](AGENT-BRIEF.md) — 规范说明文档：如何编写稳定可用的智能体任务说明
- [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md) — 规范说明文档：`.out-of-scope/` 知识库的使用规则

## 角色定义
分为两大类**分类角色**：
- `bug` 缺陷：功能运行异常
- `enhancement` 优化建议：新增功能或现有功能改进

以及五种**状态角色**：
- `needs-triage` 待分类：需维护人员评估
- `needs-info` 补充信息中：等待提交者提供更多资料
- `ready-for-agent` 待智能体处理：信息完整，可交由离线智能体执行
- `ready-for-human` 待人工处理：需要人工开发落地
- `wontfix` 不予处理：该事项不会推进整改

每一个完成分类的事项，**必须且仅能绑定一个分类角色 + 一个状态角色**。若出现状态角色冲突，先标注问题并征询维护人员意见，再执行后续操作。

以上为标准角色名称，事项平台实际标签文本可能存在差异。对应映射关系需提前配置，若未配置，请执行 `/setup-matt-pocock-skills` 完成初始化。

### 状态流转规则
无标签的新事项，默认先流转至 `needs-triage` 待分类状态；
此后可流转至 `needs-info`、`ready-for-agent`、`ready-for-human` 或 `wontfix`。
提交者补充信息后，`needs-info` 状态的事项将重新回到 `needs-triage`。

维护人员可随时手动变更状态。若发现异常流转，需先标注并确认，再继续操作。

## 调用方式
维护人员输入 `/triage` 指令，并以自然语言说明需求，系统解读指令并执行对应操作。示例：
- “查看所有需要我处理的事项”
- “查看编号#42的事项”
- “将#42改为待智能体处理状态”
- “筛选所有可交由智能体接手的事项”

## 查看待处理事项
查询事项平台，按**创建时间由早到晚**分为三类展示，并统计数量、简述每条事项内容，由维护人员选择处理目标：
1. **未打标签**：从未进行分类处置的事项
2. **待分类 needs-triage**：正在评估中的事项
3. **待补充信息 needs-info**：自上次分类后，提交者有新留言、需要重新评估的事项

## 单条事项分类处置流程
1. **梳理背景信息**
阅读事项全文、历史评论、标签、提交人及时间记录，梳理过往分类备注，避免重复询问已解决的问题。结合项目领域术语查阅代码库，遵守对应架构决策记录。查看 `.out-of-scope/` 目录下文档，核对是否存在历史驳回的同类需求/缺陷。

2. **给出处置建议**
明确推荐的分类角色与状态角色，并说明理由，同时简述代码库相关情况。等待维护人员指示。

3. **缺陷复现（仅 bug 类事项）**
开展深度问询前，先尝试复现问题：按照提交者描述的步骤操作、追踪对应代码、运行测试或相关命令，并反馈结果：成功复现（附带代码链路）、复现失败、信息不足（建议标记为 `needs-info`）。问题成功复现后，才能编写完善的智能体任务说明。

4. **深度信息问询（按需执行）**
若事项内容不完善，调用 `/grill-with-docs` 流程补充信息。

5. **执行最终处置**
- `ready-for-agent` 待智能体处理：参照 [AGENT-BRIEF.md](AGENT-BRIEF.md) 发布智能体任务说明评论。
- `ready-for-human` 待人工处理：沿用智能体任务说明的格式，同时注明无法交由智能体处理的原因（需要主观判断、依赖外部权限、涉及架构设计、需人工测试等）。
- `needs-info` 补充信息中：使用下方模板发布分类备注。
- `wontfix` 不予处理（缺陷类）：礼貌说明原因，随后关闭事项。
- `wontfix` 不予处理（优化建议类）：将内容归档至 `.out-of-scope/` 目录，在评论中附上链接，随后关闭事项（参考 [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)）。
- `needs-triage` 待分类：打上对应角色标签；若已有部分进展，可选择性补充评论。

## 快速变更状态
若维护人员直接下达指令（如“将#42改为待智能体处理状态”），直接按指令修改标签。先确认待执行操作（变更标签、发布评论、关闭事项），再执行操作，跳过信息问询环节。
若未经过信息问询就设置为 `ready-for-agent`，需询问对方是否需要补充编写智能体任务说明。

## 补充信息模板
```markdown
## 分类处置备注

**目前已确认信息：**
- 要点1
- 要点2

**还请提交者(@提交人账号)补充以下内容：**
- 问题1
- 问题2
```

已通过问询确认的内容，统一整理至「目前已确认信息」板块，避免信息丢失。提问要求具体、可落地，禁止笼统表述“请补充更多信息”。

## 继续历史处置流程
若该事项已有过往分类备注，先阅读备注内容，检查提交者是否答复了遗留问题，更新现状后再继续处置。**不再重复询问已解决的问题**。
